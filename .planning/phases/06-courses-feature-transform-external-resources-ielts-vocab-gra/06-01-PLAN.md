<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → POST /api/courses/ingest | Untrusted PDF file upload crosses here — file type, content, and size must be validated |
| PDF text → AI prompt | Extracted PDF text becomes part of the AI system prompt — could contain adversarial content |
| IndexedDB → CoursePlayer | Client-side storage is trusted but not tamper-proof — validate course data on render |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Tampering | PDF upload endpoint | mitigate | Validate magic bytes (%PDF) and file extension (.pdf) in server/routes/courses.js before processing. Reject non-PDF files with 400. |
| T-06-02 | Manipulation | AI prompt injection via PDF content | mitigate | Sanitize PDF text before injecting into AI prompt: strip control characters, limit length, escape special characters. Do not pass raw PDF text as-is into system prompts. |
| T-06-03 | Denial of Service | Large PDF upload | mitigate | Limit upload to 10MB via express.json({ limit: '10mb' }). Process PDFs synchronously (they're small enough). |
| T-06-04 | Information Disclosure | IndexedDB course data | accept | Course data is user-local only. No cross-user exposure possible with single-store IndexedDB pattern. |
| T-06-05 | Tampering | Course data in IndexedDB | mitigate | validateCourse() from courseSchema.js runs on every course load to ensure data integrity. |
| T-06-SC | Tampering | npm/pip/cargo installs | mitigate | No new packages installed. All dependencies (pdfjs-dist, tesseract.js, cheerio) already approved in Phase 1. |
</threat_model>
